# 회원가입 테이블 및 소셜 로그인 플로우

> Phase1: 소셜 로그인 가입 테이블 | Phase2: 일반 회원가입 테이블 | Phase3: 소셜 로그인 동작 플로우

---

## Phase1: 소셜 로그인 가입 시 사용하는 테이블

소셜(카카오) 회원가입 완료 시 **users**, **social_logins**, **user_roles** 만 저장된다. (주소는 수집·저장하지 않음)

### 1) users (공통 회원 테이블)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| id | BIGINT (PK) | 회원 식별자 |
| email | VARCHAR(255) | 이메일 (회원가입 폼에서 입력, 유일) |
| password | VARCHAR(255) | 암호화 비밀번호 (소셜은 랜덤 문자열 암호화) |
| name | VARCHAR(50) | 이름 (회원가입 폼에서 입력) |
| phone | VARCHAR(20) | 휴대폰 (회원가입 폼에서 입력, 유일) |
| status | VARCHAR | ACTIVE 등 |
| terms_agreed | BOOLEAN | 서비스 약관 동의 |
| privacy_agreed | BOOLEAN | 개인정보 동의 |
| terms_agreed_at | TIMESTAMP | 약관 동의 시각 |
| privacy_agreed_at | TIMESTAMP | 개인정보 동의 시각 |
| deleted_at | TIMESTAMP | 삭제 시각 (null이면 미삭제) |
| created_at | TIMESTAMP | 생성 시각 (BaseTimeEntity) |
| updated_at | TIMESTAMP | 수정 시각 (BaseTimeEntity) |

### 2) social_logins (소셜 로그인 연동)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| id | BIGINT (PK) | 식별자 |
| user_id | BIGINT (FK → users.id) | 연동된 회원 ID |
| provider | VARCHAR | `KAKAO` |
| provider_user_id | VARCHAR(255) | 카카오 회원 번호 (고유) |
| connected_at | TIMESTAMP | 연동 시각 |
| deleted_at | TIMESTAMP | 삭제 시각 (null이면 미삭제) |
| created_at | TIMESTAMP | 생성 시각 |
| updated_at | TIMESTAMP | 수정 시각 |

- **유일 제약:** (provider, provider_user_id) 조합은 중복 불가.

소셜 가입 완료 시 `KakaoService.completeKakaoSignup()` 에서 **User 1건 + SocialLogin 1건 + UserRole(CUSTOMER)** 만 생성된다. (주소 미저장)

## Phase2: 일반 회원가입 시 사용하는 테이블

일반 회원가입은 **users** 테이블에만 회원 정보를 저장하고, **user_roles**, (선택에 따라) **addresses** 등이 추가될 수 있다. **social_logins** 는 사용하지 않는다.

### users (일반 회원가입)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| id | BIGINT (PK) | 회원 식별자 |
| email | VARCHAR(255) | 이메일 (아이디, 유일) |
| password | VARCHAR(255) | 암호화 비밀번호 (가입 시 입력) |
| name | VARCHAR(50) | 이름 |
| phone | VARCHAR(20) | 휴대폰 (유일) |
| status | VARCHAR | ACTIVE 등 |
| terms_agreed | BOOLEAN | 서비스 약관 동의 |
| privacy_agreed | BOOLEAN | 개인정보 동의 |
| terms_agreed_at | TIMESTAMP | 약관 동의 시각 |
| privacy_agreed_at | TIMESTAMP | 개인정보 동의 시각 |
| deleted_at | TIMESTAMP | 삭제 시각 |
| created_at | TIMESTAMP | 생성 시각 |
| updated_at | TIMESTAMP | 수정 시각 |

- 일반 회원가입 시 `AuthService.register()` 에서 **User 1건 + UserRole(CUSTOMER) 1건** 생성.
- **social_logins** 에는 행이 없음 → 소셜 로그인 ID는 null(없음)으로 봐도 됨.

## Phase3: 소셜 로그인 동작 플로우

카카오 로그인은 **최초 로그인**과 **재로그인(이미 가입된 회원)** 으로 나뉜다.

### 3-1. 소셜 로그인 최초 로그인 플로우

1. 사용자가 프론트에서 “카카오 로그인” 클릭 → `GET /oauth2/authorization/kakao` 요청.
2. Spring Security가 카카오 인증 페이지로 리다이렉트.
3. 사용자가 카카오에서 로그인·동의 후, 카카오가 `GET /login/oauth2/code/kakao?code=...` 로 콜백.
4. Spring OAuth2가 code로 액세스 토큰·사용자 정보 조회 후 `OAuth2LoginSuccessHandler.onAuthenticationSuccess()` 호출.
5. 핸들러에서 카카오 `id` → `providerUserId`, 프로필에서 `nickname` 추출.
6. `KakaoService.isKakaoUserRegistered(providerUserId)` 호출  
   → **social_logins** 에 (KAKAO, provider_user_id) 존재 여부 확인.
7. **없음(최초)** 이면:
   - 유저 생성하지 않음.
   - 세션에 `kakao_pending_provider_user_id`, `kakao_pending_nickname` 저장.
   - 프론트로 **리다이렉트:** `{frontendSuccessUrl}?kakao=signup_required`.
8. 프론트(예: App.jsx)에서 `kakao=signup_required` 이면 회원가입 모달을 **social-extra** 모드로 오픈.
9. 사용자가 이름·이메일·휴대폰·약관 등 입력 후 “가입 완료” → `POST /api/auth/social-signup/complete` (credentials 포함). (주소 미수집)
10. 백엔드에서 세션의 `kakao_pending_provider_user_id` 로 `KakaoService.completeKakaoSignup(providerUserId, request)` 호출.
11. **User 1건 + SocialLogin(KAKAO, provider_user_id) 1건 + UserRole** 생성 후, JWT 발급·쿠키 설정하여 로그인 완료.

### 3-3. 소셜 최초 회원가입 시 DB 저장 시점

| 시점 | API/처리 | DB 저장 내용 | 비고 |
|------|----------|--------------|------|
| 카카오 콜백 직후 (최초 로그인) | `OAuth2LoginSuccessHandler` | **없음** | 세션에만 `kakao_pending_provider_user_id`, `kakao_pending_nickname` 저장. 유저/테이블 생성 안 함. |
| 회원가입 폼 제출 | `POST /api/auth/social-signup/complete` → `KakaoService.completeKakaoSignup()` | **users** 1건, **user_roles** 1건(CUSTOMER), **social_logins** 1건(KAKAO+provider_user_id) | 이 시점에만 DB 저장. 이메일/휴대폰 중복·약관 등 검증 후 한 트랜잭션으로 저장. 주소 미저장. |

- **정리**: 소셜 최초 로그인 시 DB에 유저를 만들지 않고, **필수 가입 요소(이메일 중복 확인, 휴대폰 인증, 약관 동의)를 모두 입력한 뒤** `POST /api/auth/social-signup/complete` 호출 시점에만 `users`, `social_logins`, `user_roles` 테이블에 저장된다. (주소는 수집·저장하지 않음)

**DB 저장 시점 확인 방법 (어느 테이블에서 보는지)**

| 확인 목적 | 테이블 | 확인 방법 |
|-----------|--------|-----------|
| 회원이 **한 번만** 그 시점에 생성되는지 | **users** | 카카오 최초 콜백 직후에는 해당 카카오 id로 된 새 행 없음. `social-signup/complete` 호출 후에만 `email`/`name`/`phone` 등이 들어간 새 행 1건 생김. |
| 소셜 연동이 **complete 시점에만** 들어가는지 | **social_logins** | 최초 로그인 직후에는 `provider='KAKAO'`, `provider_user_id=해당_카카오id` 인 행이 없음. complete 호출 후에만 해당 행 1건 추가됨. |
| 역할 부여 시점 | **user_roles** | 새 user_id에 대한 CUSTOMER 역할 행은 complete 호출 시에만 생성됨. |

즉, **users**, **social_logins**, **user_roles** 세 테이블에서 “회원가입 폼 제출 → social-signup/complete 호출” 시점에만 새 행이 생기는지로 확인할 수 있다.

### 3-2. 소셜 로그인 재로그인 플로우 (이미 가입된 회원)

1. 사용자가 “카카오 로그인” 클릭 → `GET /oauth2/authorization/kakao`.
2. 카카오 인증 후 `GET /login/oauth2/code/kakao?code=...` 콜백.
3. `OAuth2LoginSuccessHandler` 에서 `providerUserId` 추출.
4. `KakaoService.isKakaoUserRegistered(providerUserId)` → **social_logins** 에 (KAKAO, provider_user_id) **있음**.
5. `KakaoService.findOrCreateByKakaoInfo(providerUserId, nickname)` 호출  
   → 이미 있으므로 **기존 User** 조회만 하고, 생성하지 않음.
6. 해당 User로 `CustomUserDetails` 생성 후 `SecurityContextHolder` 에 인증 설정.
7. 프론트로 **리다이렉트:** `{frontendSuccessUrl}?kakao=success`.
8. 프론트에서 `kakao=success` 이면 `GET /api/auth/me` (credentials 포함) 로 로그인 상태 반영.

## 요약

| 구분 | 테이블 | 비고 |
|------|--------|------|
| 소셜 로그인 가입 | users, **social_logins**, user_roles | 카카오 ID는 social_logins.provider_user_id (주소 미저장) |
| 일반 회원가입 | users, user_roles (등) | social_logins 행 없음 → 소셜 ID 없음 |
| 최초 소셜 로그인 | 유저 생성 안 함 → signup_required → 회원가입 폼 제출 시 User + SocialLogin 생성 | |
| 재로그인 | social_logins 로 기존 User 조회 → 로그인 처리 → kakao=success | |
