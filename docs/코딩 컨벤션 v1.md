| 구분 | 기술 | 버전 |

### 1.1 프로젝트 폴더 구조

- **백엔드**와 **프론트엔드**는 `Final-Project` 루트 아래 **`backend`** 폴더, **`frontend`** 폴더는 새로운 루트로 구분한다.

```
Final-Project/
├── backend/          # Spring Boot (Java 21, Spring Boot 3.5.x)
├── docs/             # 프로젝트 문서 (ERD, 유스케이스, 코딩 컨벤션 등)
└── README.md         # (선택) 프로젝트 루트 README

Final-Front           # Vite + React
```

- **backend**: API 서버, DB 연동, 비즈니스 로직
- **frontend**: 웹 클라이언트 (Vite + React) (vercel로 배포하기 위해 backend와 분리하여 구성)
- **docs**: ERD, 유스케이스, 코딩 컨벤션 등 공유 문서

## 2. Git 브랜치 전략 (Git Flow)

### 2.1 브랜치 종류

| 브랜치 | 설명 | 보호 여부 |
| --- | --- | --- |
| `main` | 프로덕션 배포용, 항상 안정된 상태 유지 | ✅ Protected |
| `develop` | 다음 릴리스 개발 통합 브랜치 | ✅ Protected |
| `feature/*` | 새 기능 개발 | - |
| `bugfix/*` | develop 기준 버그 수정 | - |
| `hotfix/*` | main 기준 긴급 수정 | - |
| `release/*` | 릴리스 준비 (버전 태그, 문서 등) | - |

### 2.2 브랜치 네이밍

```
feature/기능명-이슈번호    예: feature/order-api-GM-101
bugfix/버그설명-이슈번호   예: bugfix/fix-login-validation-GM-102
hotfix/수정내용            예: hotfix/fix-payment-error
release/버전               예: release/v1.0.0
```

- 기능/버그 설명은 **영문 소문자**, 단어는 **하이픈(-)** 으로 구분
- 구조: {타입}/{간단 설명}-{이슈번호}

### 2.3 브랜치 흐름

```
main ─────●─────────────────────────●────────  (배포)
           \                       /
            \  hotfix/*           /
             \                   /
develop ──────●────●────●────●───●────
                   \   /   \   /
                    feature(bug)/*
```

- **기능 개발**: `develop` 에서 `feature/xxx` 생성 → 개발 후 `develop` 으로 PR
- **버그 수정**: `develop` 에서 `bugfix/xxx` 생성 → 수정 후 `develop` 으로 PR
- **긴급 수정**: `main` 에서 `hotfix/xxx` 생성 → 수정 후 `main`, `develop` 둘 다 PR
- **릴리스**: `develop` 에서 `release/v1.0.0` 생성 → 검증 후 `main` 머지 및 태그

### 2.4 머지 컨벤션

- **PR 머지 전**: 최소 2명 이상 승인, 충돌 해결 완료.
- **머지 방식**: 팀에서 하나로 통일 후 일괄 적용.

| 방식 | 설명 | 사용 시기 |
| --- | --- | --- |
| **Merge commit** | PR 브랜치를 그대로 머지, 머지 커밋 생성 | 브랜치 단위 이력 보존이 필요할 때 |
- **보호 브랜치**: `main`, `develop` 에는 **직접 push 금지**. 반드시 **feature/bugfix/hotfix/release** 브랜치에서 PR로만 반영.
- **머지 후**: 머지된 **feature/bugfix/hotfix** 브랜치는 삭제(origin)
- **release** → `main` 머지 후에는 **태그** 생성 (예: `v1.0.0`) 권장.

## 3. Java / Spring Boot 컨벤션

### 3.1 폴더/패키지 구조

폴더 기본 구조
- **도메인 기준 패키지 구조**
- 기술 계층 기준 분리 금지

```
domain
 └─ order
    ├─ controller
    ├─ service
    ├─ repository
    ├─ entity
    └─ dto
        ├─ request
        └─ response
```

- 패키지명: **전부 소문자**
- 파일명: PascalCase
- 공통 로직은 `global` 패키지에서만 허용

### 3.1.1 도메인 기준 패키지

- **도메인(기능) 단위**로 패키지를 나누어 유지보수와 협업 범위를 명확히 한다.
- **공통(global)**은 루트에 두고, **도메인별** 클래스는 하위 패키지로 구분한다.

| 방식 | 설명 | 예시 |
| --- | --- | --- |
| **레이어 기준** (기본) | config, domain, repository, service, controller, dto, exception 등 | 현재 3.1 트리 구조 |
| **dto 패키지** | dto/request·response 안에 도메인별 하위 패키지 | `dto/request/`, `dto/response/` |
| **도메인 패키지** | 도메인별로 controller·service·repository 묶기 | `order/`, `store/`, `user/` 각각에 Controller, Service 등 |
- **도메인 목록**: 주문(order), 마트(store), 사용자(user), 상품(product), 배달(delivery) 등 ERD·유스케이스 기준으로 통일.

### 3.2 파일명

| 유형 | 규칙 | 예시 |
| --- | --- | --- |
| Java 클래스 | PascalCase | `OrderService.java`, `OrderRequestDto.java` |
| 테스트 클래스 | 대상명 + `Test` | `OrderServiceTest.java` |
| 설정 파일 | kebab-case | `application.yml`, `application-dev.yml` |

### 3.3 클래스명

| 유형 | 규칙 | 예시 |
| --- | --- | --- |
| 엔티티 | 명사, PascalCase | `Order`, `Store`, `OrderItem` |
| 서비스 | 명사 + `Service` | `OrderService`, `StoreService` |
| 컨트롤러 | 명사 + `Controller` | `OrderController`, `StoreController` |
| DTO | 메소드 + 도메인 + (액션) + 응답or요청 | `PostUserSignUpRequest` |
| Repository | 엔티티명 + `Repository` | `OrderRepository`, `StoreRepository` |
| 예외 클래스 | 설명 + `Exception` | `OrderNotFoundException`, `InvalidRequestException` |

### 3.4 변수명 / 필드명

- **camelCase** 사용
- 의미 있는 이름 사용, 약어는 최대한 의미가 드러나도록 사용 (필요시 주석 추가)

```java
// Good
Long orderId;
String storeName;
List<OrderItem> orderItems;

// Avoid
Long oid;
String s_nm;
List<OrderItem> list;
```

- 상수: **UPPER_SNAKE_CASE**

```java
public static final int MAX_PAGE_SIZE = 20;
public static final String DEFAULT_SORT = "createdAt";
```

### 3.5 메서드명

- **camelCase**
- 동사로 시작 (조회: get/find, 생성: create, 수정: update, 삭제: delete)

```java
// Good
public Order findById(Long id)
public void createOrder(OrderRequestDto dto)
public void updateOrderStatus(Long orderId, OrderStatus status)
public boolean existsByStoreId(Long storeId)
```

- boolean 반환: `is~`, `has~`, `can~`, `exists~` 등

```java
boolean isEmpty();
boolean hasPermission();
boolean existsByEmail(String email);
```

### 3.6 컨트롤러/API 규칙

- REST 리소스는 **복수 명사** 사용
- HTTP 메서드: GET(조회), POST(생성), PUT/PATCH(수정), DELETE(삭제)

```java
@RestController
@RequestMapping("/api/v1/orders")
public class OrderController {

    @GetMapping("/{orderId}")
    public ResponseEntity<OrderResponseDto> getOrder(@PathVariable Long orderId) { }

    @PostMapping
    public ResponseEntity<OrderResponseDto> createOrder(@RequestBody OrderRequestDto dto) { }

    @PatchMapping("/{orderId}/status")
    public ResponseEntity<Void> updateOrderStatus(...) { }
}
```

### 3.7 코드 서식

- 들여쓰기: **스페이스 4칸** (또는 팀 합의 시 2칸 일괄)
- 한 줄 길이: **120자 이내** 권장, 초과 시 줄바꿈
- 중괄호: K&R 스타일 (같은 줄에 `{`)

```java
if (condition) {
    doSomething();
} else {
    doOther();
}
```

### 3.7.1 임포트 의존성 컨벤션

- **와일드카드 금지**: `import java.util.*;` 사용하지 않음. 사용하는 클래스만 명시.

```java
// Good
import java.time.LocalDateTime;
import java.util.List;
import jakarta.validation.Valid;
import org.springframework.stereotype.Service;
import com.example.neighborhoodmarket.domain.entity.Order;

// Avoid
import java.util.*;
import com.example.neighborhoodmarket.domain.entity.*;
```

- **불필요한 import 제거**: 사용하지 않는 import는 삭제. (IDE의 “Optimize Imports” 활용)

### 3.8 DTO 구조 및 컨벤션 - 예시 추가하기

### 3.8.1 Request DTO 어노테이션

- **Jakarta Validation** (`jakarta.validation.*`)으로 입력 검증을 수행한다.
- **Jackson**은 기본 스네이크/캐멀 전략만 정하고, 필드명은 **camelCase**로 통일한다.

| 어노테이션 | 용도 | 예시 |
| --- | --- | --- |
| `@NotNull` | null 불가 | ID, 필수 식별자 |
| `@NotBlank` | null/빈 문자열 불가 | 이름, 이메일 등 |
| `@NotEmpty` | null/빈 컬렉션 불가 | `List<OrderItemRequest>` |
| `@Size(min, max)` | 길이 제한 | `@Size(min = 1, max = 100)` |
| `@Min`, `@Max` | 숫자 범위 | 가격, 수량 |
| `@Email` | 이메일 형식 | 이메일 필드 |
| `@Pattern` | 정규식 | 전화번호, 주문번호 등 |
| `@Valid` | 중첩 객체 검증 | `List<@Valid OrderItemRequest>` |
- 클래스 기반이면 **Setter 비공개** 또는 **생성자 바인딩**만 사용하고, 불변에 가깝게 유지한다.

### 3.8.2 Response DTO 어노테이션

- API 스펙에서 필드 누락이 필요하면 `@JsonInclude(JsonInclude.Include.NON_NULL)` 적용 (선택).

| 어노테이션 | 용도 |
| --- | --- |
| `@JsonInclude(JsonInclude.Include.NON_NULL)` | null 필드 응답 제외 (선택) |
| `@JsonProperty("name")` | JSON 키 이름 지정 (스펙과 다를 때만) |
| `@JsonFormat` | 날짜/시간 포맷 (예: `yyyy-MM-dd'T'HH:mm:ss`) |

### 3.8.3 페이징 DTO

- **요청**: @PageableDefault 사용한다.

```java
public ResponseEntity<ApiResponse<Page<GetPaymentListResponse>>> getPaymentList(
            @RequestParam(required = false) PaymentStatus status,
            @RequestParam(required = false) String keyword,
            @PageableDefault(page = 0, size = 4, sort = "createdAt", direction = Sort.Direction.DESC) Pageable pageable,
            @AuthenticationPrincipal Long userId) {}
```

- **응답**: 앱 내의 웹 설정을 통해서 통일한다.

```java
{
    "success": true,
    "code": "SUCCESS",
    "message": "요청이 성공했습니다.",
    "data": {
        "content": [
```

### 3.9 Exception / ErrorCode 컨벤션

### 3.9.1 예외 패키지 구조

```
exception/
├── GlobalExceptionHandler.java   # @RestControllerAdvice
├── ErrorCode.java                # 에러 코드 enum
├── ApiErrorResponse.java         # API 에러 응답 공통 DTO
└── BusinessException.java    # 비즈니스 예외 (4xx)
```

### 3.9.2 ErrorCode 정의

- **enum**으로 에러 코드·메시지·HTTP 상태를 한 곳에서 관리한다.

```java
package com.example.neighborhoodmarket.exception;

import org.springframework.http.HttpStatus;

@Getter
@RequiredArgsConstructor
public enum ErrorCode {
    // 공통
    INVALID_INPUT_VALUE(HttpStatus.BAD_REQUEST, "NM001", "입력값이 올바르지 않습니다"),
    INTERNAL_SERVER_ERROR(HttpStatus.INTERNAL_SERVER_ERROR, "NM000", "서버 오류가 발생했습니다"),

    // 인증/인가
    UNAUTHORIZED(HttpStatus.UNAUTHORIZED, "NM101", "인증이 필요합니다"),
    FORBIDDEN(HttpStatus.FORBIDDEN, "NM102", "접근 권한이 없습니다"),

    // 도메인별 (주문)
    ORDER_NOT_FOUND(HttpStatus.NOT_FOUND, "NM201", "주문을 찾을 수 없습니다"),
    ORDER_ALREADY_CANCELLED(HttpStatus.BAD_REQUEST, "NM202", "이미 취소된 주문입니다"),
    ORDER_STATUS_NOT_MODIFIABLE(HttpStatus.BAD_REQUEST, "NM203", "해당 상태로 변경할 수 없습니다"),

    // 마트
    STORE_NOT_FOUND(HttpStatus.NOT_FOUND, "NM301", "마트를 찾을 수 없습니다"),
    STORE_NOT_APPROVED(HttpStatus.BAD_REQUEST, "NM302", "승인된 마트가 아닙니다");

    private final HttpStatus status;
    private final String code;   // 클라이언트 식별용 코드 (예: NM201)
    private final String message;
}
```

- **코드 체계**: 도메인_000 (예: `STORE_001`).

### 3.9.3 비즈니스 예외 클래스

- **ErrorCode**를 받아 **HTTP 상태**와 **메시지**를 일치시킨다.

```java
package com.example.neighborhoodmarket.exception.custom;

import com.example.neighborhoodmarket.exception.ErrorCode;
import lombok.Getter;

@Getter
public class BusinessException extends RuntimeException {
    private final ErrorCode errorCode;

    public BusinessException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }

    public BusinessException(ErrorCode errorCode, String overrideMessage) {
        super(overrideMessage != null ? overrideMessage : errorCode.getMessage());
        this.errorCode = errorCode;
    }
}
```

- 서비스/도메인에서: `throw new BusinessException(ErrorCode.ORDER_NOT_FOUND);`

### 3.9.4 GlobalExceptionHandler

- `@RestControllerAdvice` + `@ExceptionHandler`로 위 예외를 잡고 **ApiErrorResponse**로 반환한다.

```java
package com.example.neighborhoodmarket.exception;

import jakarta.servlet.http.HttpServletRequest;
import jakarta.validation.ConstraintViolationException;
import lombok.extern.slf4j.Slf4j;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

import java.time.Instant;
import java.util.List;
import java.util.stream.Collectors;

@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ApiErrorResponse> handleBusinessException(
            BusinessException e,
            HttpServletRequest request) {
        ErrorCode ec = e.getErrorCode();
        ApiErrorResponse body = ApiErrorResponse.builder()
                .code(ec.getCode())
                .message(e.getMessage())
                .status(ec.getStatus().value())
                .path(request.getRequestURI())
                .timestamp(Instant.now())
                .build();
        return ResponseEntity.status(ec.getStatus()).body(body);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiErrorResponse> handleValidationException(
            MethodArgumentNotValidException e,
            HttpServletRequest request) {
        List<ApiErrorResponse.FieldError> errors = e.getBindingResult().getFieldErrors().stream()
                .map(err -> ApiErrorResponse.FieldError.builder()
                        .field(err.getField())
                        .value(err.getRejectedValue() != null ? err.getRejectedValue().toString() : null)
                        .reason(err.getDefaultMessage())
                        .build())
                .collect(Collectors.toList());
        ApiErrorResponse body = ApiErrorResponse.builder()
                .code(ErrorCode.INVALID_INPUT_VALUE.getCode())
                .message(ErrorCode.INVALID_INPUT_VALUE.getMessage())
                .status(ErrorCode.INVALID_INPUT_VALUE.getStatus().value())
                .path(request.getRequestURI())
                .timestamp(Instant.now())
                .errors(errors)
                .build();
        return ResponseEntity.badRequest().body(body);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiErrorResponse> handleException(Exception e, HttpServletRequest request) {
        log.error("Unhandled exception", e);
        ErrorCode ec = ErrorCode.INTERNAL_SERVER_ERROR;
        ApiErrorResponse body = ApiErrorResponse.builder()
                .code(ec.getCode())
                .message(ec.getMessage())
                .status(ec.getStatus().value())
                .path(request.getRequestURI())
                .timestamp(Instant.now())
                .build();
        return ResponseEntity.status(ec.getStatus()).body(body);
    }
}
```

- **정리**: `BusinessException` → ErrorCode 기반 응답, `MethodArgumentNotValidException` → INVALID_INPUT_VALUE + `errors`, 그 외 → INTERNAL_SERVER_ERROR.

### 3.10 엔티티 설계

### 3.10.1 엔티티 설계 원칙

- **테이블명**: 복수형 **스네이크 케이스** (예: `orders`, `store_business_hours`).
- **컬럼명**: **스네이크 케이스** (예: `order_id`, `created_at`, `order_number`).
- **JPA 엔티티 클래스명**: **PascalCase**, 단수형 (예: `Order`, `StoreOrder`).
- **엔티티 필드**: **camelCase** → JPA가 스네이크로 매핑 (`@Column(name = "order_id")` 필요 시만 명시).
- **PK**: `Long` 타입, `id`
- **Audit**: 생성/수정 시각은 **BaseEntity**로 공통화 (아래 참고).

### 3.10.2 BaseEntity (공통 Audit)

- 모든 엔티티가 상속하여 `created_at`, `updated_at` (및 필요 시 `deleted_at`)을 공통으로 가진다.

```java
package com.example.neighborhoodmarket.domain.entity;

import jakarta.persistence.Column;
import jakarta.persistence.EntityListeners;
import jakarta.persistence.MappedSuperclass;
import lombok.Getter;
import org.springframework.data.annotation.CreatedDate;
import org.springframework.data.annotation.LastModifiedDate;
import org.springframework.data.jpa.domain.support.AuditingEntityListener;

import java.time.LocalDateTime;

@Getter
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {
    @CreatedDate
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @LastModifiedDate
    @Column(name = "updated_at", nullable = false)
    private LocalDateTime updatedAt;
}
```

- Soft Delete 사용 시 `BaseEntity` 또는 별도 `SoftDeletableEntity`에 `deletedAt` 추가하고 `@Where(clause = "deleted_at IS NULL")` 적용한다.

### 3.10.3 엔티티 클래스 예시

- **테이블명** = `orders`, **PK** = `order_id` 기준 예시이다.

```java
package com.example.neighborhoodmarket.domain.entity;

import jakarta.persistence.*;
import lombok.*;

import java.math.BigDecimal;
import java.time.LocalDateTime;
import java.util.ArrayList;
import java.util.List;

@Entity
@Table(name = "orders")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@AllArgsConstructor
@Builder
public class Order extends BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "order_id")
    private Long id;

    @Column(name = "order_number", unique = true, nullable = false, length = 50)
    private String orderNumber;

    @Column(name = "user_id", nullable = false)
    private Long userId;

    @Enumerated(EnumType.STRING)
    @Column(name = "order_type", nullable = false, length = 20)
    private OrderType orderType;

    @Enumerated(EnumType.STRING)
    @Column(name = "status", nullable = false, length = 30)
    private OrderStatus status;

    @Column(name = "total_product_price", precision = 12, scale = 0)
    private BigDecimal totalProductPrice;

    @Column(name = "total_delivery_fee", precision = 10, scale = 0)
    private BigDecimal totalDeliveryFee;

    @Column(name = "final_price", precision = 12, scale = 0)
    private BigDecimal finalPrice;

    @Column(name = "delivery_address", length = 500)
    private String deliveryAddress;

    @Column(name = "delivery_request", length = 200)
    private String deliveryRequest;

    @Column(name = "ordered_at")
    private LocalDateTime orderedAt;

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
    @Builder.Default
    private List<StoreOrder> storeOrders = new ArrayList<>();

    public enum OrderType { REGULAR, SUBSCRIPTION }
    public enum OrderStatus { PENDING, PAID, PARTIAL_CANCELLED, CANCELLED, COMPLETED }
}
```

- **네이밍**: `@Table(name = "orders")`, `@Column(name = "order_id")` 등으로 DB 컬럼명을 명시하고, 필드명은 **camelCase** 유지.
- **enum**: `@Enumerated(EnumType.STRING)` 사용.
- **연관관계**: 필요 시 `@OneToMany`, `@ManyToOne` 등으로 매핑하고, **연관 엔티티만 참조** (불필요한 FK 중복 필드 지양).

### 3.10.4 ERD 및 도메인 참고

- 실제 테이블·컬럼·관계는 **`docs/erd/ERD.md`** 를 기준으로 한다.
- 사용자: `users`, `addresses`, `social_logins`
- 마트: `stores`, `store_business_hours`
- 상품: `categories`, `products`
- 주문: `orders`, `store_orders`, `order_items` 등

엔티티 추가/수정 시 ERD와 동기화하고, **컬럼명·타입·제약조건**을 ERD에 맞춘다.

### 3.11 API / Controller

### 3.11.1 URL 규칙

- **Base path**: `/api` (버전 포함하지 않음)
- **리소스**: **복수 명사**, 소문자, 단어 구분은 **하이픈(-)** (kebab-case)

```
GET    /api/v1/orders           # 목록
GET    /api/v1/orders/{id}      # 단건
POST   /api/v1/orders           # 생성
PUT    /api/v1/orders/{id}      # 전체 수정
PATCH  /api/v1/orders/{id}      # 일부 수정 (상태 등)
DELETE /api/v1/orders/{id}      # 삭제
```

- **하위 리소스**: 부모 리소스 뒤에 이어서 표현

```
GET    /api/v1/orders/{orderId}/items
POST   /api/v1/stores/{storeId}/products
PATCH  /api/v1/orders/{orderId}/status
```

### 3.11.2 HTTP 메서드 및 역할

| 메서드 | 용도 | 응답 본문 (성공 시) |
| --- | --- | --- |
| GET | 조회 (단건/목록) | 200 + 데이터 |
| POST | 생성 | 201 + 생성된 리소스 (또는 Location 헤더) |
| PUT | 전체 수정 | 200 + 수정된 리소스 |
| PATCH | 부분 수정 (상태 변경 등) | 200 + 수정된 리소스 또는 204 No Content |
| DELETE | 삭제 | 204 No Content (또는 200 + 삭제 결과) |
- **idempotent**: GET, PUT, DELETE는 동일 요청 시 동일 결과.
- **PATCH**는 부분 필드만 받고, 상태 변경 등에 사용.

### 3.11.3 Controller 작성 규칙

- **`@RestController`** + **`@RequestMapping("/api/리소스")`** 로 공통 prefix 적용.
- **`ResponseEntity<ApiResponse<T>>`** 로 HTTP 상태와 본문을 명시적으로 반환.
- **서비스만 호출**하고, 비즈니스 로직은 서비스에 두며, DTO ↔︎ 엔티티 변환은 서비스 또는 전용 Mapper에서 처리.

```java
@RestController
@RequestMapping("/api/v1/orders")
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;

    @GetMapping("/{orderId}")
    public ResponseEntity<OrderDetailResponse> getOrder(@PathVariable Long orderId) {
        OrderDetailResponse response = orderService.getOrder(orderId);
        return ResponseEntity.ok(response);
    }

    @PostMapping
    public ResponseEntity<OrderDetailResponse> createOrder(
            @RequestBody @Valid OrderCreateRequest request) {
        OrderDetailResponse response = orderService.createOrder(request);
        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }

    @PatchMapping("/{orderId}/status")
    public ResponseEntity<Void> updateOrderStatus(
            @PathVariable Long orderId,
            @RequestBody @Valid OrderStatusUpdateRequest request) {
        orderService.updateOrderStatus(orderId, request);
        return ResponseEntity.noContent().build();
    }

    @DeleteMapping("/{orderId}")
    public ResponseEntity<Void> cancelOrder(@PathVariable Long orderId) {
        orderService.cancelOrder(orderId);
        return ResponseEntity.noContent().build();
    }
}
```

- **쿼리 파라미터**: 목록/검색 시 `@RequestParam` 또는 **Pageable** 사용. 파라미터명은 **camelCase**.
- **경로 변수**: `@PathVariable` 이름은 엔티티 ID와 맞춤 (예: `orderId`, `storeId`).

### 3.11.4 API 버전 및 공통 prefix

- CORS, 인증 등은 **config**에서 공통 적용.

### 3.13 상태값/삭제 정책 컨벤션

### 3.13.1 BaseEntity – Audit 필드 필수

- **모든 엔티티**는 **BaseEntity**를 상속해 아래 두 필드를 **필수**로 가진다.

| 필드 (엔티티) | DB 컬럼 | 설명 |
| --- | --- | --- |
| `createdAt` | `created_at` | 최초 생성 시각 (nullable=false, updatable=false) |
| `updatedAt` | `updated_at` | 마지막 수정 시각 (nullable=false) |
- **JPA Auditing** 사용: `@EnableJpaAuditing` + `@EntityListeners(AuditingEntityListener.class)` (BaseEntity에 적용).
- **타입**: `LocalDateTime` (또는 팀 합의 시 `Instant`).
- BaseEntity 정의는 [3.10.2 BaseEntity (공통 Audit)](about:blank#3102-baseentity-%EA%B3%B5%ED%86%B5-audit) 참고.

### 3.13.2 삭제 정책 – Soft Delete vs Hard Delete

- **테이블별**로 **Soft Delete** 또는 **Hard Delete** 중 하나를 정하고, ERD·엔티티·서비스에서 동일하게 적용한다.

| 정책 | 설명 | DB | 조회/삭제 처리 |
| --- | --- | --- | --- |
| **Soft Delete** | 논리 삭제. 행을 물리 삭제하지 않고 `deleted_at` 등으로 표시 | `deleted_at` (nullable) 컬럼 추가 | 조회 시 `WHERE deleted_at IS NULL`, 삭제 시 `deleted_at = NOW()` |
| **Hard Delete** | 물리 삭제. DB에서 행 자체를 삭제 | 해당 없음 | `DELETE` 또는 JPA `remove()` |

**Soft Delete 적용 테이블 (권장)**

- **사용자·주문 등 복구/감사가 필요한 데이터**: Soft Delete 권장.

| 테이블 | 삭제 정책 | 비고 |
| --- | --- | --- |
| `users` | **Soft Delete** | `deleted_at` 사용, 로그인/조회 시 제외 |
| `orders` | **Soft Delete** (선택) | 이력 보존 시 soft, 단순 주문만 있으면 hard도 가능 |
| `stores` | **Soft Delete** (선택) | 마트 폐업 이력 보존 시 |

**Hard Delete 적용 테이블 (권장)**

- **마스터/연관 데이터 중 이력 보존이 불필요한 경우**: Hard Delete.

| 테이블 | 삭제 정책 | 비고 |
| --- | --- | --- |
| `addresses` | **Hard Delete** | 사용자별 배송지 삭제는 물리 삭제 |
| `store_business_hours` | **Hard Delete** | 영업시간 행 삭제 |
| `social_logins` | **Hard Delete** (선택) | 연동 해제 시 삭제 |
| `order_items` 등 일부 자식 테이블 | **Hard Delete** (선택) | 부모(orders)가 soft면 자식은 cascade 또는 hard 팀 합의 |
- **정리**:
    - **BaseEntity**는 `createdAt`, `updatedAt` **필수**.
    - **Soft Delete**가 필요한 테이블만 `deleted_at`을 추가하고, 해당 엔티티에 `@Where(clause = "deleted_at IS NULL")` 적용.
    - 나머지는 **Hard Delete**로 일관 적용.

### 3.13.3 Soft Delete 구현 시 공통 사항

- **Soft Delete용 엔티티**는 `BaseEntity`를 확장해 `deletedAt`을 넣거나, 별도 `SoftDeletableEntity`를 만들어 상속한다.
- **컬럼**: `deleted_at` (타입: `LocalDateTime` 또는 `Instant`), nullable, 기본값 NULL.
- **조회**: `@Where(clause = "deleted_at IS NULL")` 로 삭제된 행 제외.
- **삭제 처리**: 서비스에서 해당 엔티티의 `deletedAt`만 설정하고 `save()` 호출. `delete()` 호출하지 않음.

### 4.2 파일명

## 5. Git 커밋 컨벤션

### 5.1 커밋 메시지 형식

```
<type>: <subject>

[optional body]

[optional footer]
```

- **type**: 커밋 유형 (아래 표 참고)
- **subject**: 한 줄 요약, **50자 이내**, 명령형 현재형, 마침표 생략
- 형식

```
type: 상세 요약 (#12)
```

### 5.2 Type 종류

| Type | 설명 | 예시 |
| --- | --- | --- |
| `feat` | 새 기능 | feat(order): 주문 생성 API 추가 |
| `fix` | 버그 수정 | fix(auth): 로그인 만료 시 리다이렉트 수정 |
| `docs` | 문서만 변경 | docs: README 설치 방법 보완 |
| `style` | 코드 의미 변경 없음 (포맷, 세미콜론 등) | style: Prettier 적용 |
| `refactor` | 리팩터링 (기능 변경 없음) | refactor(store): StoreService 메서드 분리 |
| `test` | 테스트 추가/수정 | test(order): OrderService 단위 테스트 추가 |
| `chore` | 빌드, 설정, 패키지 등 | chore: Spring Boot 3.5.1 업그레이드 |

### 5.3 커밋 메시지 예시

```
feat: 주문 상태 변경 API 추가
fix: 장바구니 모달 닫기 시 스크롤 복원
refactor(store): StoreRepository 쿼리 메서드 정리
```

### 5.4 규칙 요약

- **제목**: 한글/영문 가능, 팀 합의 후 통일
- **이슈 번호**: 사용 시 제목 끝에 `#105` 형태로 명시

## 6. Pull Request 컨벤션

### 6.1 PR 제목

- 커밋 메시지와 유사하게 작성 권장
예: `feat: 주문 생성 API (#101)`

### 6.2 PR 본문 (템플릿 예시)

```markdown
## 변경 사항 요약
-

-[ ] 로컬에서 빌드/단위 테스트 통과
-[ ] (해당 시) API 문서/스펙 반영
-[ ] 자기 코드 리뷰 필요 여부
```

### 6.3 리뷰 규칙

- PR은 **최소 2명 이상** 승인 후 머지
- `main`, `develop` 머지는 **Merge commit** 일괄 적용
