# 소셜 로그인 백로그 검증 보고

> Phase1~Phase5 검증 결과 정리 (백로그, 로그, 코딩 컨벤션, API 명세서, DB 저장 시점)

---

## Phase1: POST social-login / social-register 등 기능상 구현 필요 여부

**결론: 기능상 구현할 필요 없다.**

- 현재 **OAuth2 리다이렉트 + POST /api/auth/social-signup/complete** 로 이미 다음을 만족한다.
  - **POST /api/auth/social-login** (provider, accessToken): OAuth 검증·기존/신규 구분은 Spring OAuth2 콜백에서 code → 토큰·사용자 정보 조회로 동일하게 수행됨. 별도 토큰 기반 API 없이도 동작.
  - **POST /api/auth/social-register** (신규 유저 전용): 동일 목적을 **POST /api/auth/social-signup/complete** 가 수행함(세션의 pending + 필수 가입 요소 검증 후 User·SocialLogin·JWT 생성).
  - **신규 유저(최초 소셜 로그인)는 social_logins에 해당 provider_user_id 없음**: 최초 콜백 시 유저를 생성하지 않고 세션에만 pending 저장하므로, DB의 `social_logins`에는 complete 호출 전까지 해당 provider_user_id가 없음. 플로우상 이미 보장됨.
  - **JWT 발급/응답에 isNewUser 포함**: 현재는 complete 성공 시에만 로그인되므로 “이 요청 = 신규 가입 완료”로 한정됨. 명세서 형태의 JSON에 isNewUser를 넣지 않아도 기능상 문제 없음. (나중에 명세와 응답 형식을 맞출 때 추가 가능.)

따라서 **토큰 기반 API(POST social-login / social-register)와 isNewUser 응답은 기능상 필수는 아니며, 현재 구현만으로 요구 사항을 충족한다.**  
다만 **동네마켓_API_명세서_v4** 또는 외부 클라이언트(앱 등)에서 **토큰 기반** 소셜 로그인 API를 요구하면 그때 해당 API를 추가 구현하면 된다.

## Phase1(상세): 백로그 대비 구현 검증

### User Story 대비

| 요구 사항 | 구현 여부 | 비고 |
|-----------|-----------|------|
| 카카오/네이버 소셜 계정 로그인 | ⚠️ 카카오만 적용 | 네이버 미구현 |
| OAuth 토큰 검증 후 기존 연동/신규 가입 처리 | ✅ | Spring OAuth2 Client로 code → 토큰·사용자 정보 조회 후 처리 |
| 신규 시 필수 가입 요소 충족 전 정식 JWT 미발급 | ✅ | 최초 로그인 시 signup_required 리다이렉트, `POST /api/auth/social-signup/complete` 성공 시에만 JWT 발급 |

### 할 일 목록(Tasks) 대비

| 항목 | 상태 | 비고 |
|------|------|------|
| `POST /api/auth/social-login` (provider, accessToken) | ❌ 미구현 | 명세는 **Body로 provider+accessToken** 받는 API. 현재는 **OAuth2 리다이렉트**(GET /oauth2/authorization/kakao → callback) 방식만 구현. |
| `POST /api/auth/social-register` (신규 유저 전용) | ❌ 별도 경로 | 동일 목적을 **POST /api/auth/social-signup/complete** 로 구현(세션 기반). |
| OAuth 토큰 검증, AccessToken으로 사용자 정보 조회 | ✅ | Spring OAuth2가 code → accessToken → `/v2/user/me` 호출로 처리. |
| social_logins 동일 provider+socialId 있으면 기존 유저 로그인 | ✅ | `KakaoService.isKakaoUserRegistered()` → `findOrCreateByKakaoInfo()` |
| 없으면 신규 → 가입 폼 후 User+SocialLogin 생성 | ✅ | 세션에 pending 저장 후 `completeKakaoSignup()` 에서 users, social_logins, user_roles, addresses 저장 |
| 신규 유저 필수 가입(이메일 중복, 휴대폰 인증, 약관 등) | ✅ | `SocialSignupCompleteRequest` 검증 + 서비스에서 이메일/휴대폰 중복·약관 검사 |
| JWT 응답에 isNewUser 포함 | ❌ | 명세에는 isNewUser 필드 요구. 현재 complete 응답은 MeResponse(userId, email, name, roles)만 반환. |

### 완료 조건(Acceptance Criteria) 대비

| # | 조건 | 구현 여부 |
|---|------|-----------|
| 1 | 정상 요청 시 200 OK, accessToken, refreshToken, isNewUser 포함 | ⚠️ 기존 회원 로그인은 리다이렉트+세션으로 JWT는 프론트에서 /api/auth/me 등으로 확인. complete 시 201+Set-Cookie+MeResponse (body에 accessToken/refreshToken 없음, 쿠키로 전달). |
| 2 | 신규 소셜 유저 OAuth 검증 성공 시 202, 정식 JWT 미발급 | ✅ | 리다이렉트 `?kakao=signup_required` 로 전달. 202 HTTP가 아닌 **302 리다이렉트**로 동일 의도 구현. |
| 3 | 신규가 필수 가입 완료 후 POST social-register 성공 시 200 OK 응답 | ✅ | **POST /api/auth/social-signup/complete** 성공 시 **201 CREATED** + Set-Cookie(JWT) + MeResponse. (명세는 200, 컨벤션상 생성은 201 사용.) |
| 4 | OAuth 토큰 검증 실패 시 401 | ✅ | Spring OAuth2/카카오 연동 실패 시 인증 실패 처리. |
| 5 | 이미 다른 계정 연동된 이메일 시 409 Conflict | ✅ | `completeKakaoSignup` 에서 `DUPLICATE_EMAIL`(409) 발생. |
| 6 | 신규 가입 완료 시 역할 CUSTOMER, social_logins 저장 | ✅ | `completeKakaoSignup` 에서 UserRole(CUSTOMER), SocialLogin 저장. |

**요약**: 비즈니스 요구(최초 로그인 시 가입 폼 → 필수 정보 입력 후 한 번에 DB 저장·JWT 발급)는 **리다이렉트 + POST /api/auth/social-signup/complete** 로 충족. 다만 **API 경로·방식**은 백로그/명세의 `POST /api/auth/social-login`, `POST /api/auth/social-register` 와 다름.

## Phase2: 백엔드 로그

다음 위치에 로그를 추가하여 플로우를 백엔드 로그에서 확인할 수 있도록 함.

| 위치 | 로그 내용 |
|------|-----------|
| `OAuth2LoginSuccessHandler` | `[OAuth2 카카오] 기존 회원 로그인 처리 providerUserId=...` / `[OAuth2 카카오] 최초 로그인 - 회원가입 폼 필요 providerUserId=...` |
| `KakaoService.isKakaoUserRegistered` | (debug) `[카카오] providerUserId=... 등록 여부=...` |
| `KakaoService.completeKakaoSignup` | `[카카오 소셜 가입] 시작 providerUserId=..., email=...` / 이메일·휴대폰 중복·이미 가입 시 warn / `[카카오 소셜 가입] 완료 userId=..., providerUserId=...` |
| `AuthController.completeSocialSignup` | `[소셜 가입 완료] API 호출 providerUserId=...` / 세션 없을 때 `[소셜 가입 완료] 세션에 kakao_pending_provider_user_id 없음` (warn) |

| 항목 | 결과 | 비고 |
|------|------|------|
| 도메인 기준 패키지 (auth) | ✅ | auth 하위에 controller, service, dto, config 등 |
| Base path /api | ✅ | `/api/auth` 사용 (버전 미포함) |
| ResponseEntity\<ApiResponse\<T\>\> | ✅ | 컨트롤러 일괄 적용 |

## Phase4: 동네마켓_API_명세서_v4 위배 여부

| 항목 | 명세 | 현재 구현 | 위배 여부 |
|------|------|-----------|-----------|
| 소셜 로그인 API | **POST /api/auth/social-login**, Body: provider, accessToken(필수), email, name(선택) | **미구현**. OAuth2 리다이렉트 방식만 존재. | ✅ 위배 (API 미구현) |
| 소셜 로그인 응답 | 200 OK, data 내 accessToken, refreshToken, isNewUser, userId, email, name, roles | 기존 회원은 리다이렉트 후 세션; JWT는 쿠키. 응답 Body 형태 불일치. | ✅ 위배 |
| 신규 유저 202 응답 | (백로그 기준) 신규 시 202, JWT 미발급 | 302 리다이렉트 + signup_required. 202 미사용. | 명세서에는 202 조건 없음. 백로그와만 상이. |
| POST /api/auth/social-register | 백로그에만 명시, 명세서 목차에는 없음 | **POST /api/auth/social-signup/complete** 로 구현 | 명세서에 social-register 없음 → 경로 상이만 기록 |

**요약**: **동네마켓_API_명세서_v4** 의 **API-AUTH-003 (POST /api/auth/social-login)** 는 **토큰 기반 요청·명시적 JSON 응답**을 전제로 하나, 현재는 **리다이렉트 기반 + POST /api/auth/social-signup/complete** 만 구현되어 있어 **명세와 구현 방식이 다름**. 명세를 따르려면 `POST /api/auth/social-login`(및 필요 시 `POST /api/auth/social-register`) API를 추가 구현해야 함.

## Phase5: 소셜 최초 회원가입 시 DB 저장 시점

**문서**: `회원가입_테이블_및_소셜로그인_플로우.md` 에 **「3-3. 소셜 최초 회원가입 시 DB 저장 시점」** 섹션 추가함.

### 요약 표

| 시점 | API/처리 | DB 저장 |
|------|----------|---------|
| 카카오 콜백 직후 (최초 로그인) | OAuth2LoginSuccessHandler | **없음** (세션에만 pending 저장) |
| 회원가입 폼 제출 | POST /api/auth/social-signup/complete → KakaoService.completeKakaoSignup() | **users**, **user_roles**, **social_logins**, **addresses** 한 트랜잭션으로 저장 |

- **정리**: DB에 유저를 넣는 시점은 **오직** `POST /api/auth/social-signup/complete` 가 호출되어 `KakaoService.completeKakaoSignup()` 이 실행될 때뿐이다. 그 전까지는 어떤 회원 테이블에도 저장하지 않는다.

## 권장 후속 작업

1. **API 명세·백로그와의 정합성**:  
   - 옵션 A) 명세대로 **POST /api/auth/social-login**(provider, accessToken), **POST /api/auth/social-register** 를 추가하고, 기존 리다이렉트 플로우는 유지하거나 점진 이전.  
   - 옵션 B) 현재 리다이렉트 + social-signup/complete 방식을 공식 플로우로 확정하고, 명세서를 이 방식에 맞게 수정(경로·응답·상태 코드 반영).

2. **예외 처리**: `KakaoService` 의 `IllegalStateException` 을 **ErrorCode + BusinessException** 으로 교체 검토.

3. **응답 필드**: 소셜 로그인/가입 완료 응답에 **isNewUser** 포함 여부를 명세와 맞춤.
