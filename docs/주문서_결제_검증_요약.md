# 주문서·결제 관련 백엔드 검증 요약

> 주문서 미리보기(Checkout) 조회 API, 주문 상세 조회 API, 금액 계산 공용화, 알림 연동 포인트 검증 및 수정 사항 정리.

---

## Phase1: 생성 파일 목록 확인 ✅

백로그 기준 제시된 파일 모두 존재함.

| 구분 | 경로 |
|------|------|
| Checkout | `checkout/controller/CheckoutController`, `checkout/service/CheckoutService`, `checkout/dto/response/GetCheckoutResponse`, `checkout/service/PriceCalculator`, `DefaultPriceCalculator`, `PriceCalculationResult` |
| Order | `order/controller/OrderController`, `order/dto/response/GetOrderDetailResponse`, `order/service/OrderQueryService`, `order/repository/OrderRepository`, `StoreOrderRepository`, `OrderProductRepository`, `CartProductRepository` |
| Notification | `communication/service/NotificationService`, `communication/service/OrderPaidNotificationService`, `communication/service/impl/DbNotificationService` |

---

## Phase2: 파일 형식·관계 연결 ✅

- **CheckoutController** → CheckoutService → PriceCalculator(DefaultPriceCalculator), CartProductRepository, AddressRepository, PaymentMethodRepository 등 의존성 정상.
- **OrderController** → OrderQueryService → OrderRepository, StoreOrderRepository, OrderProductRepository, PaymentRepository 정상.
- **GetCheckoutResponse**: 주소, 결제 초기값, 마트별 그룹(상품·수량·재고·판매상태·금액), PriceSummary(상품총액/배달비/할인/포인트/최종결제) 포함.
- **GetOrderDetailResponse**: order, storeOrders(products), payment 포함. 결제 완료 화면/영수증용으로 사용 가능.
- **PriceCalculator**: 상품총액·배달비·할인·포인트 반영한 최종 결제 금액 계산 공용화. BR-O03 동일 입력 시 동일 결과.

---

## Phase3: 코딩 컨벤션 v1 반영 ✅

적용한 수정:

1. **API 응답 포맷**  
   - 컨벤션 3.11.3, 3.12: `ResponseEntity<ApiResponse<T>>` 사용.  
   - `CheckoutController`, `OrderController`가 **ApiResponse.success(message, data)** 로 래핑하여 반환하도록 수정.

2. **에러 코드**  
   - 주문 없음 시 `ErrorCode.NOT_FOUND` → **ORDER_NOT_FOUND** (ORDER-001) 사용.  
   - `ErrorCode`에 `ORDER_NOT_FOUND` 추가, `OrderQueryService`에서 사용.

3. **PaymentRepository 쿼리**  
   - `Payment` 엔티티는 `Order` 연관관계만 있고 `orderId` 필드는 없음.  
   - `findByOrderId` → **findByOrder_Id(Long orderId)** 로 변경 (Spring Data JPA 속성 경로).

4. **패키지/네이밍**  
   - 도메인 기준 패키지( checkou t, order, communication ), DTO 네이밍( GetCheckoutResponse, GetOrderDetailResponse ), 복수 리소스 `/api/checkout`, `/api/orders` 유지.

---

## Phase4: 엔티티·프론트 기준 동작 ✅

- **Order**: `user`, `orderNumber`, `status`, `totalProductPrice`, `totalDeliveryFee`, `finalPrice`, `deliveryAddress`, `deliveryRequest`, `orderedAt` 등. `GetOrderDetailResponse`와 매핑 가능.
- **StoreOrder / OrderProduct**: 마트별 주문, 상품 스냅샷(이름·단가·수량) 제공. 영수증/상세 화면 렌더링 가능.
- **Payment**: `paymentStatus`, `amount`, `paymentMethod`, `receiptUrl` 등. 결제 정보 노출에 사용.
- **Address**: `addressName`, `contact`, `addressLine1`, `addressLine2` → GetCheckoutResponse.AddressInfo와 일치.

---

## Phase5: 프론트 연동 (연결 포인트)

- **공통 응답**: 모든 성공 응답이 `{ success, code, message, data, timestamp }` 형태.  
  - **실제 payload는 `data` 필드**에 있음.  
  - 주문서 미리보기: `response.data` → `GetCheckoutResponse` (address, payment, storeGroups, priceSummary).  
  - 주문 상세: `response.data` → `GetOrderDetailResponse` (order, storeOrders, payment).

- **엔드포인트**
  - 주문서 미리보기: `GET /api/checkout?cartItemIds={id1,id2,...}&addressId={optional}`  
    - 인증: Bearer(또는 쿠키).
  - 주문 상세: `GET /api/orders/{orderId}`  
    - 인증: Bearer(또는 쿠키).

- **프론트 미구현**: 현재 프론트에서 `/api/checkout`, `/api/orders` 호출 코드는 없음.  
  - 결제 페이지/결제 완료 페이지 연동 시 위 엔드포인트를 호출하고 **`response.data`** 를 사용하면 됨.

## Phase6: 로깅 ✅

- **CheckoutController**: `getCheckout` 요청 시 `email`, `cartItemIds`, `addressId` debug 로그.
- **CheckoutService**: cartItemIds 비어 있을 때 warn, 정상 처리 시 `userId`, `storeGroups.size()` debug 로그.
- **OrderController**: `getOrderDetail` 요청 시 `email`, `orderId` debug 로그.
- **OrderQueryService**: 정상 조회 시 `orderId`, `userId` debug 로그.

로그 레벨을 `debug` 이상으로 설정하면 백엔드 로그에서 위 플로우 추적 가능.

## 알림 연동 (결제 성공 시)

- **인터페이스**: `communication.service.OrderPaidNotificationService`
