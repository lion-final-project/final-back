# 회원가입·로그인·로그아웃·소셜 로그인 설계 (리뷰용 상세 설명)

> 내일 리뷰를 위한 설계 정리. 흐름·API·DB·보안·클래스 역할을 중심으로 서술합니다.

---

## 1. 개요

- **일반 회원가입**: 이메일 + 비밀번호 + 휴대폰 인증 + 약관 → `users`·`user_roles` 저장 후 **즉시 JWT 발급** (자동 로그인).
- **일반 로그인**: 이메일/비밀번호 검증 후 **JWT(access + refresh)** 발급, **HttpOnly 쿠키**로 전달.
- **로그아웃**: 서버에 저장된 refresh token 삭제 + **쿠키 삭제** 응답. (토큰 없어도 200으로 처리해 프론트 항상 성공)
- **소셜 로그인(카카오)**: OAuth2로 카카오 인증 → **이미 가입된 사용자**는 바로 로그인(세션 + 리다이렉트), **최초 사용자**는 세션에 pending 저장 후 프론트 회원가입 폼 → `POST /api/auth/social-signup/complete`로 **User + SocialLogin** 생성 후 **JWT 발급**.

인증 수단은 **JWT(access/refresh)** 이며, access token으로 API 호출 시 인증되고, refresh token으로 **재발급**합니다. 소셜은 “카카오로 들어온 뒤 우리 DB에 한 번 등록”하는 회원가입 + 로그인으로 취급합니다.

## 2. 전체 아키텍처

```
[프론트 localhost:5173]
    │  withCredentials: true (쿠키 포함)
    ▼
[Vite proxy /api → 8080]
    │
    ▼
[백엔드 localhost:8080]
    ├── SecurityFilterChain: /api/admin/** 만 authenticated, 나머지 permitAll
    ├── JwtAuthenticationFilter: 쿠키(nm_accessToken) 또는 Authorization: Bearer 에서 access token 추출 → 검증 후 SecurityContext에 인증 설정
    ├── OAuth2 (카카오): /oauth2/authorization/kakao → 카카오 로그인 → 콜백 → OAuth2LoginSuccessHandler
    └── AuthController: /api/auth/* (register, login, logout, me, refresh, social-signup/complete 등)
            │
            ▼
    AuthService, KakaoService, JwtTokenProvider, RefreshTokenRepository, UserRepository ...
            │
            ▼
    DB: users, user_roles, refresh_tokens, social_logins, roles
```

- **프론트**: axios `withCredentials: true`, baseURL은 프록시로 `/api` → 8080.
- **백엔드**: Spring Security + JWT(access/refresh) + OAuth2 Client(카카오). 인증 정보는 **SecurityContext**에 담기며, **JWT**일 때는 principal이 **이메일(String)**, **OAuth 세션**일 때는 **CustomUserDetails**입니다.

## 3. 사용 기술·개념

| 항목 | 설명 |
|------|------|
| **JWT** | Access Token(짧은 유효기간)·Refresh Token(긴 유효기간). 서명 검증으로 위변조 방지. |
| **HttpOnly 쿠키** | access/refresh를 쿠키에 담아 전달. JavaScript에서 접근 불가 → XSS로 토큰 탈취 어렵게 함. |
| **Refresh Token 저장** | DB `refresh_tokens` 테이블에 저장. 로그아웃/재발급 시 해당 레코드 삭제·갱신. |
| **Spring Security** | 요청별 인증(Filter), 권한 체크(authorizeHttpRequests). OAuth2 로그인 시 SuccessHandler에서 리다이렉트·세션 처리. |
| **OAuth2 (카카오)** | 사용자를 카카오 로그인 페이지로 보냄 → 카카오가 인증 후 code로 우리 서버 리다이렉트 → 우리가 code로 토큰·사용자 정보 조회. |

## 4. DB 테이블 (인증 관련)

| 테이블 | 역할 |
|--------|------|
| **users** | 이메일, 비밀번호(암호화), 이름, 휴대폰, 약관 동의 등. 일반·소셜 모두 한 테이블. |
| **user_roles** | user_id ↔ role_id (N:N). 역할 예: CUSTOMER, ADMIN. |
| **roles** | role_name (CUSTOMER, ADMIN 등). |
| **refresh_tokens** | user_id, token, 만료 시각. 로그인 시 기존 삭제 후 새로 저장, 로그아웃 시 삭제. |
| **social_logins** | user_id, provider(KAKAO), provider_user_id. “이 카카오 ID는 이 user와 연결됨”을 저장. |

- 일반 회원가입: **users** 1건 + **user_roles** 1건(CUSTOMER). social_logins 없음.
- 소셜 최초 가입 완료: **users** 1건 + **user_roles** 1건 + **social_logins** 1건.
- 로그인/소셜 가입 완료 시: **refresh_tokens**에 refresh token 1건 저장 (동일 user 기존 건 삭제 후).

## 5. 일반 회원가입 설계

### 5.1 흐름 (단계)

1. 프론트: 이메일 중복 확인 → `GET /api/auth/check-email?email=...`
2. 프론트: 휴대폰 인증번호 발송 → `POST /api/auth/send-verification` (body: `{ phone }`)
3. 프론트: 인증번호 확인 → `POST /api/auth/verify-phone` (body: `{ phone, verificationCode }`) → 응답에 **phoneVerificationToken** 포함
4. 프론트: 약관 동의, 비밀번호 등 입력 후 → `POST /api/auth/register` (SignupRequest)
5. 백엔드: 이메일·휴대폰 중복 재확인, 휴대폰 인증 토큰 검증, 비밀번호 암호화 후 **User** 저장, **UserRole(CUSTOMER)** 저장, **JWT(access+refresh)** 발급, **refresh_tokens** 저장, **Set-Cookie**로 토큰 전달
6. 프론트: 201 + 쿠키 수신 → 로그인 상태로 전환 (추가 로그인 호출 불필요)

### 5.2 API: POST /api/auth/register

- **Request (SignupRequest)**: email, password, name, phone, phoneVerificationToken, termsAgreed, privacyAgreed, marketingAgreed 등.
- **Response**: 201, body에 SignupResponse(userId, email, name, roles, accessToken, refreshToken), **헤더에 Set-Cookie** (nm_accessToken, nm_refreshToken).
- **검증**: 이메일/휴대폰 중복, 휴대폰 인증 토큰(또는 인증 완료 상태), 비밀번호 길이 등. 실패 시 BusinessException(ErrorCode).

### 5.3 서비스 로직 (AuthService.register)

- `userRepository.existsByEmail`, `existsByPhone` → 중복 시 DUPLICATE_EMAIL / DUPLICATE_PHONE.
- `smsService.validateAndConsumeVerificationToken` 또는 `validateAndConsumePhoneVerified`로 휴대폰 인증 소비.
- `PasswordEncoder.encode(request.getPassword())`로 암호화.
- User 엔티티 저장, Role(CUSTOMER) 조회/생성, UserRole 저장.
- JWT access/refresh 생성, refresh_tokens 기존 삭제 후 저장, SignupResponse 반환.

## 6. 일반 로그인 설계

### 6.1 흐름

1. 프론트: 이메일·비밀번호 전송 → `POST /api/auth/login` (LoginRequest)
2. 백엔드: 이메일로 User 조회 → 없으면 404(EMAIL_NOT_FOUND). 비밀번호 비교(PasswordEncoder.matches) → 불일치 401(PASSWORD_MISMATCH). User 상태가 ACTIVE가 아니면 403(USER_STATUS_FORBIDDEN).
3. 백엔드: user_roles에서 roles 조회, access/refresh JWT 생성, **refresh_tokens**에서 해당 user 기존 건 삭제 후 새 refresh token 저장.
4. 응답: 200 + body(LoginResponse: userId, email, name, roles, accessToken, refreshToken) + **Set-Cookie**(nm_accessToken, nm_refreshToken).
5. 프론트: 쿠키 저장됨, 이후 API 호출 시 쿠키 자동 포함. `/api/auth/me`로 현재 사용자 정보 조회 가능.

### 6.2 API: POST /api/auth/login

- **Request**: LoginRequest { email, password }
- **Response**: 200, body + Set-Cookie (access, refresh)
- **에러**: 404 EMAIL_NOT_FOUND, 401 PASSWORD_MISMATCH, 403 USER_STATUS_FORBIDDEN

### 6.3 JWT·쿠키 설정

- **Access Token**: 유효기간 예) 30분. 쿠키 이름 `nm_accessToken`, HttpOnly, Path=/, SameSite=Lax.
- **Refresh Token**: 유효기간 예) 14일. 쿠키 이름 `nm_refreshToken`, 동일 옵션.
- **JwtAuthenticationFilter**: 요청 시 **Cookie(nm_accessToken)** 또는 **Authorization: Bearer {token}** 에서 access token 추출 → 검증 후 SecurityContext에 `UsernamePasswordAuthenticationToken(subject=email, null, authorities)` 설정. 이후 컨트롤러에서 Authentication으로 “현재 로그인 사용자” 식별.

## 7. 로그아웃 설계

### 7.1 흐름

1. 프론트: `POST /api/auth/logout` (body 없이, **credentials 포함**으로 쿠키 전송)
2. 백엔드: **Cookie(nm_refreshToken)** 또는 body의 refreshToken으로 refresh token 획득.
3. **토큰이 있으면**: DB refresh_tokens에서 해당 토큰 조회 후 **삭제** (해당 refresh로 재발급 불가).
4. **토큰이 없어도**: 400을 내지 않고 200 처리 (이미 로그아웃된 상태로 간주).
5. 응답: 200 + **Set-Cookie**로 nm_accessToken, nm_refreshToken **삭제** (maxAge=0).
6. 프론트: 로그아웃 UI로 전환.

### 7.2 API: POST /api/auth/logout

- **Request**: 쿠키(nm_refreshToken) 또는 body RefreshTokenRequest { refreshToken } (선택).
- **Response**: 200, "로그아웃 되었습니다.", Set-Cookie(삭제)
- **설계 포인트**: refresh token이 없어도 200으로 응답해, 쿠키가 안 넘어오는 환경(프록시/도메인 이슈)에서도 프론트가 항상 “로그아웃 성공”으로 처리할 수 있음.

---

## 8. 토큰 갱신 (Refresh) 설계

- **API**: `POST /api/auth/refresh`, **Cookie(nm_refreshToken)** 에서 refresh token 읽음.
- **로직**: refresh token 검증, DB에 존재·일치 여부 확인 후, 새 access·refresh 생성, DB의 refresh_tokens 레코드 갱신, 새 토큰을 다시 Set-Cookie로 내려줌.
- **용도**: access token 만료 시 프론트가 refresh 호출 → 새 access로 API 재시도. (현재 프론트에서 자동 호출 여부는 구현에 따라 다름)

## 9. 소셜 로그인(카카오) 설계

### 9.1 OAuth2 흐름 요약

1. 사용자가 “카카오 로그인” 클릭 → 브라우저가 **백엔드** `GET /oauth2/authorization/kakao` 로 이동.
2. Spring Security OAuth2 Client가 **카카오 로그인 페이지**로 리다이렉트 (client_id, redirect_uri, scope 등 포함).
3. 사용자가 카카오에서 로그인·동의 → 카카오가 **우리 백엔드** `GET /login/oauth2/code/kakao?code=...` 로 리다이렉트.
4. 백엔드가 **code**로 카카오에 토큰 요청 → **access_token** 수신 → 그 토큰으로 **카카오 사용자 정보**(id, nickname 등) 조회.
5. **OAuth2LoginSuccessHandler** 실행: 카카오 사용자 id(providerUserId)로 우리 DB **social_logins** 조회.

### 9.2 이미 가입된 카카오 회원 (재방문)

- `social_logins`에 (KAKAO, providerUserId) 존재 → “이미 우리 서비스에 연동된 사용자”.
- **KakaoService.findOrCreateByKakaoInfo(providerUserId, nickname)** 로 User·roles 조회.
- **CustomUserDetails** 생성 후 **SecurityContextHolder**에 인증 설정 (세션에 올리는 효과).
- **리다이렉트**: `{frontendSuccessUrl}?kakao=success` (예: http://localhost:5173?kakao=success).
- **JWT는 이 단계에서 발급하지 않음.** 프론트는 **같은 도메인(5173)** 에서 백엔드(8080)로 요청할 때, **세션 쿠키(JSESSIONID)** 가 넘어가므로 `/api/auth/me`를 호출하면 **세션의 CustomUserDetails**로 사용자 정보를 받을 수 있음. 단, 실제 우리 프로젝트는 **프론트가 5173, 백 8080** 이라 세션 공유가 안 될 수 있어, **실제로는** “카카오 success 리다이렉트 후 프론트에서 /api/auth/me 호출”이 **JWT가 있어야** 동작할 수 있음. → 현재 구현에서는 **기존 카카오 회원도** success 리다이렉트 후 **프론트가 /api/auth/me를 호출**하고, 이때 **access token(쿠키)** 이 있으면 그걸로 me가 반환됨. **카카오 success 시점에는 아직 우리가 JWT를 새로 발급하지 않으므로**, 기존에 로그인된 상태(쿠키)가 없으면 me는 실패할 수 있음. → **문서상** “기존 회원은 success 리다이렉트 + (가능하면) 세션 또는 기존 JWT로 /api/auth/me” 라고만 두고, 상세는 코드/테스트로 확인하는 것이 좋음.

### 9.3 최초 카카오 로그인 (미가입)

- `social_logins`에 (KAKAO, providerUserId) **없음** → “아직 우리 서비스 회원 아님”.
- **세션에 pending 저장**: `kakao_pending_provider_user_id`, `kakao_pending_nickname`.
- **리다이렉트**: `{frontendSuccessUrl}?kakao=signup_required`.
- **JWT 미발급.** 사용자는 “추가 정보 입력” 화면으로 보내짐.

### 9.4 회원가입 완료 (추가 정보 입력 후)

1. 프론트: `kakao=signup_required` 인 경우 회원가입 모달(social-extra) 표시, 이름·이메일·휴대폰·휴대폰 인증·약관 입력.
2. **동일 세션**으로 `POST /api/auth/social-signup/complete` 호출 (SocialSignupCompleteRequest).  
   - **세션에서** `kakao_pending_provider_user_id` 를 꺼내야 하므로, **같은 브라우저·같은 백엔드 세션**이 유지되어야 함 (프록시 사용 시 5173→8080 요청이 같은 세션으로 가야 함).
3. **AuthController.completeSocialSignup**: 세션에서 providerUserId 조회, 없으면 400. **KakaoService.completeKakaoSignup(providerUserId, request)** 호출.
4. **KakaoService.completeKakaoSignup**:
   - 이미 (KAKAO, providerUserId) 있으면 예외.
   - 이메일·휴대폰 중복 검사.
   - **User** 1건 생성 (비밀번호는 랜덤 문자열 암호화), **UserRole(CUSTOMER)** 1건, **SocialLogin(KAKAO, providerUserId)** 1건 저장.
5. **AuthService.issueTokensForUser**: 해당 User로 **access/refresh JWT** 생성, **refresh_tokens** 저장, **Set-Cookie** 설정.
6. 응답: **201** + Set-Cookie(access, refresh) + body(MeResponse: userId, email, name, roles).
7. 세션에서 `kakao_pending_provider_user_id`, `kakao_pending_nickname` 제거.
8. 프론트: 로그인 상태로 전환.

### 9.5 API: POST /api/auth/social-signup/complete

- **Request**: SocialSignupCompleteRequest { name, email, phone, termsAgreed, privacyAgreed } (휴대폰 인증 등은 프론트에서 선검증 가능).
- **전제**: **세션에 kakao_pending_provider_user_id 가 있어야 함** (카카오 최초 로그인 직후).
- **Response**: 201, Set-Cookie(access, refresh), body MeResponse.
- **에러**: 이메일/휴대폰 중복(409 등), 세션에 pending 없음(400).

## 10. 현재 사용자 조회: GET /api/auth/me

- **인증**: JwtAuthenticationFilter가 **access token(쿠키 또는 Bearer)** 으로 SecurityContext에 인증을 채움. principal이 **이메일(String)**(JWT) 또는 **CustomUserDetails**(OAuth 세션).
- **AuthService.getCurrentUser(authentication)**:
  - **CustomUserDetails**면: user, authorities에서 roles 추출 → MeResponse.
  - **String(email)**이면: userRepository.findByEmail + user_roles 조회 → MeResponse.
- **Response**: 200, MeResponse { userId, email, name, roles }.
- **용도**: 앱 로딩 시·카카오 success 리다이렉트 후 “지금 로그인한 사람이 누구인지” 확인, 역할에 따른 UI 분기.

## 11. 보안·설정 요약

| 항목 | 내용 |
|------|------|
| 비밀번호 | BCrypt 등으로 암호화 저장, 로그인 시 matches로 비교. |
| JWT | 서명 검증, 만료 시간 검사. access는 짧게, refresh는 길게. |
| 쿠키 | HttpOnly, SameSite=Lax. XSS로 토큰 읽기 어렵게. |
| Refresh Token | DB 보관, 로그아웃/재발급 시 삭제·갱신. 재사용 방지. |
| CORS | local 개발 시 5173, 3000 등 허용, credentials true. |
| 인증 필요 경로 | 예: /api/admin/** 만 authenticated, /api/auth/* 대부분 permitAll(login, register, refresh 등). |

## 12. 주요 클래스 역할

| 클래스 | 역할 |
|--------|------|
| **AuthController** | /api/auth/* 엔드포인트: register, login, logout, me, refresh, check-email, check-phone, send-verification, verify-phone, social-signup/complete. DTO 받아서 서비스 호출, Set-Cookie 설정. |
| **AuthService** | register, login, logout, refresh, getCurrentUser, issueTokensForUser, 휴대폰 인증 발송/검증. User·UserRole·RefreshToken 저장/조회. |
| **KakaoService** | isKakaoUserRegistered(providerUserId), findOrCreateByKakaoInfo, completeKakaoSignup(providerUserId, request). 카카오 API(토큰·사용자 정보) 호출, User·SocialLogin·UserRole 저장. |
| **OAuth2LoginSuccessHandler** | 카카오 로그인 성공 후: providerUserId로 기존 가입 여부 판단, 기존이면 SecurityContext 설정 + kakao=success 리다이렉트, 신규면 세션 pending + kakao=signup_required 리다이렉트. |
| **JwtAuthenticationFilter** | 매 요청: 쿠키 또는 Bearer에서 access token 추출 → 검증 → SecurityContext에 인증 설정. |
| **JwtTokenProvider** | access/refresh 생성, 검증, claims 파싱 (subject=email, roles). |
| **CookieUtil** | nm_accessToken, nm_refreshToken 쿠키 생성/삭제 (Path, HttpOnly, MaxAge, SameSite). |
| **CustomUserDetails** | User + roles를 Spring Security UserDetails로 감싼 것. getAuthorities()에서 ROLE_ 접두어. OAuth success 시 SecurityContext에 넣을 때 사용. |
| **CustomUserDetailsService** | (일반 로그인 폼 사용 시) 이메일로 User 조회 후 CustomUserDetails 반환. 현재는 JWT 로그인 위주라 로그인 엔드포인트에서 직접 AuthService 호출. |

## 13. 에러 코드 (인증 관련)

| 코드 | HTTP | 의미 |
|------|------|------|
| EMAIL_NOT_FOUND | 404 | 로그인 시 해당 이메일 없음. |
| PASSWORD_MISMATCH | 401 | 비밀번호 불일치. |
| USER_STATUS_FORBIDDEN | 403 | 정지/비활성 등 로그인 불가 상태. |
| DUPLICATE_EMAIL / DUPLICATE_PHONE | 409 등 | 회원가입·소셜 가입 시 중복. |
| REFRESH_TOKEN_MISSING | 400 | 로그아웃 시 refresh 없음 (현재는 200으로 처리하도록 완화). |
| REFRESH_TOKEN_INVALID | 401 | refresh 만료·위조·DB에 없음. |
| UNAUTHORIZED | 401 | /api/auth/me 등에서 인증 없음. |

## 14. 프론트 연동 요약

- **baseURL**: 프록시로 `/api` → 8080. **withCredentials: true** 필수.
- **회원가입**: check-email → send-verification → verify-phone → register. 201 수신 시 쿠키 저장, 로그인 상태로 전환.
- **로그인**: login 호출, 200 + 쿠키. 이후 me 호출해 userId, roles 등 저장.
- **로그아웃**: logout 호출, 200 + 쿠키 삭제. 로컬 상태 초기화.
- **카카오**: 버튼 클릭 시 `window.location = /oauth2/authorization/kakao` (백엔드 주소). 리다이렉트 돌아온 뒤 **쿼리** `kakao=success` / `kakao=signup_required` 에 따라 me 호출 또는 회원가입 모달(social-extra) → social-signup/complete 호출.

이 문서는 “회원가입·로그인·로그아웃·소셜 로그인”이 **어떤 흐름으로 연결되고, 어떤 API·DB·클래스가 쓰이는지** 리뷰할 때 참고할 수 있도록 정리한 설계 설명입니다. 세부 검증(휴대폰 인증 Redis 키, 카카오 success 시 JWT 발급 여부 등)은 코드와 실제 호출로 확인하면 됩니다.
