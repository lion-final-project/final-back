| 백로그 Task | 구현 여부 | 구현 위치 (백엔드) |
|-------------|-----------|---------------------|
| **회원가입 입력값 검증** | ✅ | `SignupRequest` (@Valid), `AuthController.register` |

| 인증번호 생성 및 저장 | ✅ | `SmsUtil.generateAuthCode()` 6자리, **Redis** `SMS:AUTH:{phone}` 저장 |
| 인증번호 유효시간 3분 | ✅ | Redis TTL 3분 (`AUTH_TTL_MINUTES = 3`) |
| 재발송 횟수 제한 5회 | ✅ | Redis `SMS:RESEND:{phone}` 카운트, `RESEND_LIMIT = 5` |
| **휴대폰 인증번호 검증** | ✅ | POST /verify-phone, `AuthService.verifyPhone` → `SmsService.verifyAuthCode` |
| 인증번호 일치 여부 확인 | ✅ | Redis 저장값과 입력값 비교, 불일치 시 `PHONE_VERIFICATION_MISMATCH` |
| 인증번호 만료 처리 | ✅ | Redis 키 없음 → `PHONE_VERIFICATION_EXPIRED` |
| 인증 성공 시 상태 반영 | ✅ | Redis `SMS:VERIFIED:{token}`(10분) + **방법 1** `SMS:PHONE_VERIFIED:{phone}`(5분) 저장. 회원가입 시 토큰 또는 phone-verified 중 하나 1회 사용 후 삭제 |
| **회원 정보 저장 및 기본 권한** | ✅ | `AuthService.register` |
| 비밀번호 암호화 저장 | ✅ | `PasswordEncoder` (BCrypt), `User` 저장 |
| 회원 정보 저장 | ✅ | `userRepository.save(user)` |
| 기본 역할 부여 | ✅ | CUSTOMER Role 조회/생성, `UserRole` 저장 |
| 가입 완료 응답 반환 | ✅ | 201 Created, `SignupResponse` (userId, email, name, roles) + Set-Cookie(`nm_accessToken`, `nm_refreshToken`) |

## 협업 의사전달용 구현 요약

| 항목 | 구현 방식 | 프론트/협업 시 참고 |
|------|-----------|---------------------|
| **JWT 토큰 전달** | Access/Refresh 토큰은 **HttpOnly Cookie**로만 전달. 쿠키 이름: `nm_accessToken`, `nm_refreshToken`. 응답 본문에는 토큰 필드 없음. | 회원가입·로그인·refresh 응답에서 **Set-Cookie** 확인. API 요청 시 `credentials: 'include'` 필수. |
| **휴대폰 인증 저장** | DB 테이블 없음. **Redis** 사용: `SMS:AUTH:{phone}`(인증번호 3분), `SMS:RESEND:{phone}`(재발송 5회), `SMS:VERIFIED:{token}`(인증 완료 토큰 10분), **방법 1** `SMS:PHONE_VERIFIED:{phone}`(인증 완료 상태 5분). | 인증번호·재발송·인증완료 토큰은 모두 Redis에만 존재. |
| **인증 완료 후 회원가입** | **방법 1 적용**: `POST /verify-phone` 성공 시 (1) `phoneVerificationToken` 반환 + (2) Redis `SMS:PHONE_VERIFIED:{phone}` TTL 5분 저장. 회원가입 시 **토큰 있으면** 토큰 검증, **토큰 없으면** 5분 이내 `SMS:PHONE_VERIFIED` 검증. 둘 다 1회 사용 후 삭제. | 5분 이내면 `phoneVerificationToken` 없이 register만 호출해도 됨(교차검증·테스트 유리). |
| **에러 코드** | `COMMON-xxx`, `AUTH-xxx` 형식 (예: AUTH-002 이메일 중복, AUTH-003 휴대폰 중복, AUTH-006 인증 만료, AUTH-009 인증번호 불일치). | 응답 `error.code`로 분기 처리. |
| **비밀번호** | BCrypt 암호화 저장. 검증: 8자 이상, 영문+숫자+특수문자. | 평문 전송, 저장은 서버에서 암호화. |
| **기본 역할** | 가입 시 `CUSTOMER` 역할 자동 부여. `roles` 테이블에 없으면 생성 후 `user_roles`에 저장. | JWT claim 및 응답 `data.roles`에 `["CUSTOMER"]` 포함. |

## Phase 3: GitHub 공유 시 타인 동일 동작·Role 반영

### 3.1 동일 값 출력을 위해 필요한 것

| 항목 | 설명 | 공유 시 유의사항 |
|------|------|-------------------|
| **프로파일** | `spring.profiles.active=local` | 동일하게 `local` 사용 시 동일 설정 로드 |
| **application-local.yml** | JWT secret, DB, Redis 등 | 리포지터리에 포함 시 **동일 JWT·DB 설정**으로 동일 동작. secret은 프로덕션에서는 환경변수로 교체 권장 |
| **DB** | PostgreSQL, `ddl-auto: update` | 스키마는 자동 생성. 타인도 로컬 DB(mydatabase 등) 준비 필요 |
| **Role** | CUSTOMER | `roles` 테이블에 없으면 가입 시 `Role.builder().roleName("CUSTOMER").build()`로 생성 후 `user_roles`에 저장 → 환경마다 동일하게 기본 역할 부여됨 |

### 3.2 Role 반영 흐름 (동작 문제 없음)

1. **가입 시**: `AuthService.register` → `roleRepository.findByRoleName("CUSTOMER")` → 없으면 `save(Role.builder().roleName("CUSTOMER").build())` → `UserRole(user, role)` 저장.
2. **JWT**: `SignupResponse`에 `roles: ["CUSTOMER"]` 포함, AccessToken claim에도 `roles` 포함.
3. **인증 시**: `JwtAuthenticationFilter`에서 토큰의 `roles`를 파싱해 `SimpleGrantedAuthority("ROLE_" + role)`로 설정 → Spring Security에서 `ROLE_CUSTOMER`로 인식.

따라서 GitHub에서 clone 후 **동일하게 `local` 프로파일 + application-local.yml** 사용하면, Role 포함 동작에 문제 없음.

## Phase 4: Task별 구현 플로우

### Task 1: 회원가입 입력값 검증 로직

1. 클라이언트: `POST /api/auth/register` + JSON body (`SignupRequest` 스펙).
2. `AuthController.register(@Valid @RequestBody SignupRequest)` → `@Valid`로 검증.
3. `SignupRequest`: 이메일(@Email, @Pattern), 비밀번호(8자 이상, 영문+숫자+특수), 휴대폰(@Pattern), 이름, phoneVerificationToken, termsAgreed, privacyAgreed (@AssertTrue).
4. 검증 실패 시 `MethodArgumentNotValidException` → `GlobalExceptionHandler` → 400 + 메시지.

### Task 2: 이메일/휴대폰 중복 확인 API

1. **이메일**: `GET /api/auth/check-email?email=...` → `AuthController.checkEmail` → `AuthService.isEmailDuplicated` → `UserRepository.existsByEmail` → `DuplicateCheckResponse(duplicated)` 반환.
2. **휴대폰**: `GET /api/auth/check-phone?phone=...` → `AuthController.checkPhone` → `AuthService.isPhoneDuplicated` → `UserRepository.existsByPhone` → `DuplicateCheckResponse(duplicated)` 반환.

### Task 3: 휴대폰 인증번호 발송

1. `POST /api/auth/send-verification` + `{ "phone": "01012345678" }` → `AuthService.sendVerification(phone)` → `SmsService.sendAuthCode(phone)`.
2. 이미 가입된 휴대폰이면 `DUPLICATE_PHONE` (409).
3. **Redis** `SMS:RESEND:{phone}` 카운트 증가, 5회 초과 시 `PHONE_VERIFICATION_RESEND_LIMIT` (429).
4. `SmsUtil.generateAuthCode()` 6자리 생성 → CoolSMS 발송 → **Redis** `SMS:AUTH:{phone}` 저장 (TTL 3분).
5. 응답: `SendVerificationResponse` (message, expiresIn=180, remainingAttempts).

### Task 4: 휴대폰 인증번호 검증

1. `POST /api/auth/verify-phone` + `{ "phone", "verificationCode" }` → `AuthService.verifyPhone(phone, code)` → `SmsService.verifyAuthCode(phone, code)`.
2. **Redis** `SMS:AUTH:{phone}` 없으면 `PHONE_VERIFICATION_EXPIRED` (422).
3. 저장값과 입력 불일치 → `PHONE_VERIFICATION_MISMATCH` (400).
4. 통과 시 Redis 키 삭제, **Redis** `SMS:VERIFIED:{uuid}` 저장 (phone, TTL 10분) → `phoneVerificationToken` 반환 (회원가입 시 사용).

### Task 5: 회원 정보 저장 및 기본 권한 부여

1. `POST /api/auth/register` → `AuthService.register(request)`.
2. 이메일/휴대폰 중복 재확인 → **Redis** `SMS:VERIFIED:{token}` 조회 및 `validateAndConsumeVerificationToken(phone, token)` (1회 사용 후 삭제).
3. `PasswordEncoder.encode(password)` → `User` 빌드(이메일, 암호화 비밀번호, 이름, 휴대폰, 약관 동의) → `userRepository.save`.
4. `Role` "CUSTOMER" 조회 없으면 생성 후 `UserRole(savedUser, role)` 저장.
5. `JwtTokenProvider`로 AccessToken·RefreshToken 생성 → `SignupResponse` (userId, email, name, roles; 토큰은 @JsonIgnore).
6. `AuthController`에서 201 Created + **Set-Cookie** (`nm_accessToken`, `nm_refreshToken`) + 본문 `SignupResponse` 반환.

## Phase 5: AccessToken, RefreshToken, JWT, Spring Security 진행 방식

### 5.1 전체 구조

- **인증 수단**: JWT (Access Token, Refresh Token).
- **서명 알고리즘**: HS256, SecretKey는 `security.jwt.secret` (32자 이상).
- **세션**: Stateless (서버에 세션 저장 없음).

### 5.2 JWT 설정 (JwtProperties)

- `security.jwt.secret`: 서명용 비밀키 (application-local.yml 또는 환경변수).
- `security.jwt.access-token-validity-seconds`: 1800 (30분).
- `security.jwt.refresh-token-validity-seconds`: 1209600 (14일).

### 5.3 AccessToken / RefreshToken 발급

- **발급 시점**: 회원가입 성공 시 `AuthService.register` 내부.
- **내용**: `JwtTokenProvider.generateToken(user, roles, validitySeconds)`  
  - subject: `user.getEmail()`  
  - claim: `roles` (예: `["CUSTOMER"]`)  
  - issuedAt, expiration  
  - 서명: `signWith(secretKey)` (HS256).
- AccessToken: validity = accessTokenValiditySeconds (30분).
- RefreshToken: validity = refreshTokenValiditySeconds (14일).
- **클라이언트 전달**: **Set-Cookie** 헤더로만 전달 (HttpOnly, `nm_accessToken`, `nm_refreshToken`). 응답 본문에는 토큰 미포함.

### 5.4 JWT 검증 및 인증 (Spring Security)

1. **JwtAuthenticationFilter** (OncePerRequestFilter)
   - 요청마다 `Authorization` 헤더 확인.
   - `Bearer {token}` 형태면 토큰 추출 → `JwtTokenProvider.validateToken(token)` (파싱 성공 여부).
   - 유효하면 `parseClaims(token)` → subject(이메일), roles claim 사용.
   - `UsernamePasswordAuthenticationToken(principal=이메일, authorities=ROLE_* 리스트)` 생성 후 `SecurityContextHolder`에 설정.
   - 이후 컨트롤러/서비스에서 `SecurityContext`로 인증 정보 사용 가능.

2. **SecurityConfig**
   - Stateless: `SessionCreationPolicy.STATELESS`.
   - 인증 없이 허용: GET /check-email, /check-phone, POST /register, **POST /refresh**, /send-verification, /verify-phone, /error.
   - 그 외: `anyRequest().authenticated()` → JWT 유효해야 접근.
   - `JwtAuthenticationFilter`를 `UsernamePasswordAuthenticationFilter` 앞에 추가.

### 5.5 RefreshToken 갱신 API

- **엔드포인트**: `POST /api/auth/refresh`
- **요청**: Request Body 사용 안 함. **Cookie**의 `nm_refreshToken`으로 전달 (클라이언트: `credentials: 'include'`).
- **처리**: `AuthService.refresh(refreshToken)`  
  - RefreshToken 서명·만료 검증 (`JwtTokenProvider.validateToken`)  
  - 실패 시 `REFRESH_TOKEN_INVALID` (401)  
  - 성공 시 subject(이메일)로 `User` 조회, `user_roles`에서 roles 조회  
  - 새 AccessToken·RefreshToken 발급 → **Set-Cookie**로 전달
- **응답**: 200 + **Set-Cookie** (`nm_accessToken`, `nm_refreshToken`), 본문 `data`는 null 또는 메시지.
- **용도**: AccessToken 만료 후 클라이언트가 Cookie의 RefreshToken으로 새 토큰 쌍을 받아 재인증할 때 사용.

### 5.6 흐름 요약

```
[회원가입 성공]
  → AuthService.register
  → User 저장, UserRole(CUSTOMER) 저장
  → JwtTokenProvider.generateAccessToken / generateRefreshToken
  → 201 Created + Set-Cookie(nm_accessToken, nm_refreshToken) + 본문(userId, email, name, roles)

[AccessToken 만료 시]
  → 클라이언트: POST /api/auth/refresh (Cookie: nm_refreshToken, credentials: 'include')
  → AuthService.refresh: RefreshToken 검증 → User·roles 조회 → 새 AccessToken·RefreshToken 발급
  → Set-Cookie(nm_accessToken, nm_refreshToken) 반환

[이후 API 호출]
  → 클라이언트: Authorization: Bearer {accessToken}
  → JwtAuthenticationFilter: 토큰 검증 → Claims에서 이메일·roles 추출
  → SecurityContext에 인증 정보 설정
  → 컨트롤러/서비스에서 인증된 사용자·역할 사용
```

## CoolSMS 휴대폰 인증 (application-local.yml)

- CoolSMS API 키/시크릿은 **application-local.yml**의 `coolsms.api-key`, `coolsms.api-secret`에 설정합니다.
- 실제 문자 발송이 필요하면 Solapi 콘솔에서 발급한 키로 위 값을 교체하고, `coolsms.sender`를 콘솔에 등록한 발신번호로 설정합니다.

### 휴대폰 인증 문자 수신 확인 (Phase4 목표)

1. `application-local.yml`의 `coolsms.api-key`, `coolsms.api-secret`을 Solapi 콘솔에서 발급한 값으로 교체.
2. `coolsms.sender`를 Solapi 콘솔에 등록한 발신번호로 설정.
3. 백엔드: `./gradlew bootrun` 실행.
4. 프론트: 회원가입 → 휴대폰 번호 입력 → **인증요청** 클릭.
5. 입력한 휴대폰 번호로 **[동네마켓 인증번호] XXXXXX** 문자가 도착하는지 확인.
